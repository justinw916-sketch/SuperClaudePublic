Desktop Scheduled Tasks · ready to paste

Scheduled-task recipes

The Desktop app can run a saved prompt on a schedule — the app-native replacement for cron jobs and Agent-SDK automations (which are CLI-only). Below are three ready-to-paste tasks that build on this toolkit. Copy the prompt, set the cadence, point it at a project, done.

How to add a scheduled task

  • In the Desktop app, open Scheduled tasks (via the app's scheduled-tasks feature — see the docs).
  • Create a new task, set its schedule (daily/weekly + time) and its working project (the folder it runs against).
  • Paste one of the prompts below as the task's instruction. Save.
Safety by design. These recipes are read + report by default — they never commit, push, or delete on their own. They surface findings for you to act on in a live session. The danger-check and secret-scan hooks still apply.

1 · Nightly /review sweep

Catch risky uncommitted changes before you forget what they were

Cadence: daily · 6:00 PM Scope: one active repo read-only report
Paste as the task prompt
Review today's uncommitted work in this repo and flag anything risky before I lose context.

1. Run: git status --short  and  git diff  (and git diff --staged).
   If there are no changes, reply "No uncommitted changes." and stop.
2. Apply the /review checklist: secrets, injection, unvalidated input, weak
   randomness, missing error handling, PII in logs, Windows path safety, and
   any new logic missing tests.
3. Output a single VERDICT: APPROVE | REQUEST_CHANGES | REJECT, then a short
   findings list as [BLOCKER]/[WARNING]/[NOTE] with file:line and a concrete fix each.
4. Do NOT stage, commit, push, or edit anything. Report only.
Keep it under ~250 words.

Change the time/scope to taste. If you keep several active repos, make one task each, or generalize step 1 to loop over a list of paths.

2 · Weekly build digest

A Friday summary of what shipped, what's mid-flight, and what's next

Cadence: weekly · Fri 4:00 PM Scope: a project or repo read-only report
Paste as the task prompt
Produce a one-page weekly digest for this project.

1. Summarize the last 7 days of git history: run
   git log --since="7 days ago" --pretty=format:"%h %ad %s" --date=short
   Group commits into themes (features, fixes, chores).
2. If BUILD_STATE.md exists, report the current bounded-build stage, what was
   completed, and the next stage.
3. List anything that looks unfinished or risky: TODO/FIXME added this week
   (grep -rn "TODO\|FIXME" changed files), failing or missing tests, and any
   stage stuck more than a few days.
4. End with "Top 3 for next week" as a short prioritized list.
Format: tight prose with a short bullet list only where it earns its place.
Do not modify any files.

Pipe the output wherever you like afterward — the task can also draft it as a note or message if you connect a relevant tool.

3 · Daily bounded-build resume nudge

Never lose the thread of a multi-stage build

Cadence: daily · 8:30 AM Scope: your current build repo read-only report
Paste as the task prompt
Give me a 5-line resume brief for the bounded build in this repo.

1. Read BUILD_PLAN.md and BUILD_STATE.md. If neither exists, reply
   "No bounded build in progress." and stop.
2. Report: current stage (done/in-progress), the single most important thing the
   next session must know (from the latest handoff log), and the next stage's
   deliverable + files it will touch.
3. End with the exact command to resume, e.g. "Open a fresh Code-tab session and
   run /stage N".
Do not build anything. Brief only.

Tips

  • Keep tasks read-only. Let them find and summarize; do the acting yourself in a live session where you can see the diff pane.
  • One task per repo/scope is clearer than one task that guesses which project you meant.
  • Stagger times so tasks don't all fire at once.
  • Escalate on demand: when a nightly sweep flags a BLOCKER, open the repo and run /review live to fix it with full context.
