Claude Code Toolkit
for the Desktop app (Code tab)

One app-focused configuration you install from the Plugin manager — bounded-build workflow, always-on operating rules, safety hooks, tailored subagents, and an in-session /review. Built to keep Claude sharp deep into long sessions, entirely inside the Claude Desktop app. No terminal, no claude CLI.

Why this exists

Large builds don't fail from lack of capability — they fail from context rot. Model quality degrades as the context window fills: facts in the middle get lost, tool schemas eat the budget, and late-session output gets sloppier. The toolkit's answer is to keep every working session small and bounded, push durable state to disk, and verify each step. This edition delivers all of that through the Desktop Code tab, which runs the same Claude Code engine as the CLI and reads the same ~/.claude config.

Illustrative. Each /stage runs in a fresh context, so usage stays in the healthy band instead of climbing until quality falls off.

Install in ~2 minutes (no terminal)

1
Install the plugin. In the Desktop app open the Code tab, click the + next to the prompt box → Plugins → Add plugin. Add this repo/folder as a marketplace (it contains .claude-plugin/marketplace.json), then install claude-code-toolkit. This delivers all commands, agents, skills, and hooks.
2
(Optional) Personalize your identity. The operating rules already load automatically via the plugin, so this step is optional. Open ~/.claude/CLAUDE.md in the file pane and paste the contents of reference/CLAUDE.md. The Code tab reads this on every session. Optional: copy output-styles/terse-engineer.md into ~/.claude/output-styles/ and select it. (Windows shortcut: run reference/install-memory.ps1 once — it only copies these files, it does not use the CLI.)
3
Verify. Type / in the prompt box; you should see /plan, /stage, /handoff, /verify, /review, /ship. Under + → Skills you should see bounded-build, operating-rules, review.
Windows prereq: the first time you open the Code tab you need Git for Windows installed; restart the app afterward.

The Code tab is built around panes you arrange freely. The toolkit is designed to use them:

  • subagent — watch a delegated review/verify run in its own context.

The + button

Everything installs and is invoked from here:

  • Skills — browse/invoke bounded-build, review, operating-rules (or just type /).
  • Plugins — add this marketplace, manage installs.
  • Connectors — MCP servers with a graphical setup.

The bounded-build loop

/plan <goal> writes BUILD_PLAN.md + BUILD_STATE.md and stops. Each /stage N builds one plan stage, verifies it, appends a handoff log, and stops. You /clear (or open a fresh Code-tab session) between stages so context never piles up. Disk is the source of truth — a brand-new session resumes from the two state files alone.

The single most important habit: one stage per session. If a stage feels bigger than one context window, split it in the plan first.

Auto-advance the whole build — /build-all & /build-all-auto

Running /stage 1 → /clear → /stage 2 by hand is safe but tedious. /build-all automates it without reintroducing context rot: it hands each stage to a fresh stage-runner subagent, so every stage still gets its own clean context — the subagent boundary replaces /clear. Your main session only collects short summaries.

How it runs:

  • reads BUILD_PLAN.md / BUILD_STATE.md and finds the remaining stages (start anywhere with /build-all 3);
  • runs each stage in its own stage-runner subagent, posts a 2–3 line checkpoint, and continues automatically on success;
  • halts immediately if a stage's exit criterion isn't met, it hits a blocker, or the stage is too big to fit one context — it never papers over a failure;
  • runs /verify against the acceptance criteria at the end.

Watch progress in the subagent / tasks pane. For a hands-off run, switch the app to Auto permission mode so it doesn't pause for tool approvals each stage.

Checkpoint (/build-all) vs full-auto (/build-all-auto)

Both run each stage in its own fresh-context subagent and halt on the first stage whose exit criterion isn't met — they differ only in how much they involve you as they go.

/build-all — checkpoint/build-all-auto — full-auto

Rule of thumb: start with /build-all so you see each stage as it lands; switch to /build-all-auto only once the plan is proven and you're happy to walk away. Both stop cold on a failed stage — full-auto never means "ignore failures."

Automatic (/build-all) vs manual (/stage + /clear)

Use automatic /build-all when…Drive it manually when…
the plan is solid and the stages are well-scopedthe plan is exploratory or likely to change between stages
stages are independent and low-riska stage makes irreversible or high-stakes changes (migrations, deploys, deletes)
you want to walk away and let it runyou want to review and steer after each stage
it's a repeatable build you've run beforeit's a brand-new build and you're still learning the shape of the work
Rule of thumb: manual for the first pass of an unfamiliar build (you see every stage as it happens), automatic once the plan is proven. Either way a failed stage stops the line — fix it, then re-run /build-all to resume where it left off.

Review before every commit

The Desktop Code tab can't run headless claude -p, so instead of a background git hook the toolkit reviews in your session, where you can see the diff and approve.

Run /review before committing. It:

  • reads your staged/uncommitted diff and checks it for secrets, injection, correctness, error handling, PII in logs, Windows path safety, and missing tests;
  • emits a VERDICT: APPROVE | REQUEST_CHANGES | REJECT with file:line findings and concrete fixes;
  • for risky changes, hands off to the review-agent subagent for a second opinion in a separate context (watch it in the subagent pane);
  • on approval, stages files by name and writes an imperative commit with a "why" body. It never uses --no-verify / --force unless you explicitly say so.

Recurring automation → Scheduled tasks

Anything you'd have cron'd or run via the Agent SDK on the CLI, do with the Desktop app's Scheduled tasks instead (a nightly review sweep, a weekly digest, a health check). It's the desktop-native equivalent and needs no terminal. See Scheduled tasks.

Troubleshooting

  • A new command doesn't appear after installing the plugin — restart the Code tab.
  • Windows, first Code-tab launch fails — install Git for Windows, then restart the app.
  • /agents says "isn't available" — that's a terminal-only dialog; manage subagents from Customize in the sidebar.
  • A hook seems to block something legitimate — the guards are conservative and fail-open; run the exact command yourself outside Claude Code, or adjust the pattern in hooks/.
  • MCP server won't connect on Windows — verify config, restart the app, check the process in Task Manager.
